# 分类与图标模块

## 一、功能概述

分类与图标模块负责组织任务的分类管理和图标系统，主要功能包括：

- **分类管理**：创建、编辑、删除分类，按分类组织任务
- **图标系统**：系统预设图标 + 自定义图标，支持图片上传和裁剪
- **图标缓存**：LruCache 缓存策略，优化图标加载性能

```sql
CREATE TABLE category (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  icon VARCHAR(100) DEFAULT 'default',
  color VARCHAR(20) DEFAULT '#FF6B00',
  sort_order INT DEFAULT 0,
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 2.4 分类列表查询

**查询逻辑：**

```javascript
const sql = `
    SELECT c.id, c.name, c.icon, c.color, c.sort_order, 
           DATE_FORMAT(c.create_time, '%Y-%m-%d %H:%i:%s') as create_time,
           DATE_FORMAT(c.update_time, '%Y-%m-%d %H:%i:%s') as update_time,
           COUNT(t.id) as task_count
    FROM category c
    LEFT JOIN task t ON c.id = t.category_id AND t.status = 1
    GROUP BY c.id
    ORDER BY c.sort_order ASC, c.id ASC
`;
```

**关键点：**
- `LEFT JOIN task` 统计每个分类的活跃任务数量
- `t.status = 1` 只统计活跃任务
- `GROUP BY c.id` 分组统计
- `ORDER BY sort_order ASC` 按排序字段排序

### 2.8 图标存储方案

**存储格式：**
- `icon_content`：Base64 编码的图片数据
- `icon_format`：图片格式（png, jpg, webp 等）
- `icon_type`：system（系统预设）或 custom（用户自定义）

**图标引用格式：**

```javascript
// 自定义图标 - 使用 custom:{id} 格式
```

### 2.9 图标创建逻辑

```javascript
router.post('/create', async (ctx) => {
    const { display_name, icon_content, icon_format, sort_order } = ctx.request.body;

    if (!icon_content) {
        ctx.body = { code: 400, msg: '图标内容不能为空' };
        return;
    }

    const timestamp = Date.now();
    const name = `custom_${timestamp}`;

    const sql = `
        INSERT INTO icon (name, display_name, icon_type, icon_content, icon_format, sort_order, is_active)
        VALUES (?, ?, 'custom', ?, ?, ?, 1)
    `;

    const result = await pool.query(sql, [
        name,
        display_name || '自定义图标',
        icon_content,
        icon_format || 'png',
        sort_order || 0
    ]);

    ctx.body = {
        code: 200,
        msg: '创建成功',
        data: { 
            id: result[0].insertId,
            name: name,
            reference: `custom:${result[0].insertId}`
        }
    };
});
```

**返回 reference 字段的原因：**
- 前端可以直接使用 `reference` 作为任务图标字段的值
- 格式统一，便于解析

```kotlin
private fun updateAdapter() {
    val items = mutableListOf<TaskManagerItem>()

    // 1. 分类部分
    if (categories.isNotEmpty()) {
        items.add(TaskManagerItem.SectionHeader("分类管理", categories.size, showAddButton = true))
        categories.forEach { category ->
            items.add(TaskManagerItem.CategoryItemWrapper(category))
        }
    }

    // 2. 任务部分（按分类分组）
    items.add(TaskManagerItem.SectionHeader("全部任务", allTasks.size, showAddButton = false))

    val tasksByCategory = allTasks.groupBy { it.category_id }

    // 先显示有分类的任务
    categories.forEach { category ->
        val categoryTasks = tasksByCategory[category.id] ?: emptyList()
        if (categoryTasks.isNotEmpty()) {
            categoryTasks.forEach { task ->
                items.add(TaskManagerItem.TaskItemWrapper(task))
            }
        }
    }

    // 最后显示未分类任务
    val uncategorizedTasks = tasksByCategory.filterKeys { it == null }.values.flatten()
    if (uncategorizedTasks.isNotEmpty()) {
        uncategorizedTasks.forEach { task ->
            items.add(TaskManagerItem.TaskItemWrapper(task))
        }
    }

    binding.rvCategories.adapter = CategoryAdapter(items, ...)
}
```

### 3.5 图标映射系统

**IconMapper 核心功能：**

```kotlin
object IconMapper {

    // 系统图标映射
    private val iconMap = mapOf(
        "book" to R.drawable.ic_book,
        "running" to R.drawable.ic_running,
        "study" to R.drawable.ic_study,
        "exercise" to R.drawable.ic_exercise,
        // ... 更多图标
    )

    // 缓存机制（5MB）
    private val bitmapCache = object : LruCache<String, Bitmap>(5 * 1024 * 1024) {
        override fun sizeOf(key: String, value: Bitmap): Int {
            return value.byteCount
        }
    }

    // 获取资源ID
    fun getResIdFromName(name: String): Int {
        return iconMap[name] ?: R.drawable.ic_book
    }

    // 解析图标引用
    fun parseIconReference(iconReference: String): Pair<String, String> {
        return if (iconReference.startsWith("custom:")) {
            Pair("custom", iconReference.removePrefix("custom:"))
        } else {
            Pair("system", iconReference)
        }
    }

    // Base64 解码 Bitmap
    fun loadBitmapFromBase64(base64: String): Bitmap? {
        val cached = bitmapCache.get(base64)
        if (cached != null) return cached

        return try {
            val decodedString = Base64.decode(cleanBase64, Base64.DEFAULT)
            val bitmap = BitmapFactory.decodeByteArray(decodedString, 0, decodedString.size)
            bitmapCache.put(base64, bitmap)
            bitmap
        } catch (e: Exception) {
            null
        }
    }
}
```

### 3.6 图标加载器

**IconLoader 工作流程：**

```kotlin
object IconLoader {

    // 独立的 Bitmap 缓存（5MB）
    private val bitmapCache = object : LruCache<String, Bitmap>(5 * 1024 * 1024) {
        override fun sizeOf(key: String, value: Bitmap): Int {
            return value.byteCount
        }
    }

    fun loadIcon(
        imageView: ImageView,
        iconReference: String,
        placeholderResId: Int = R.drawable.ic_book,
        scope: CoroutineScope,
        onLoaded: (Bitmap?) -> Unit = {}
    ) {
        // 1. 系统图标：直接加载资源
        if (IconMapper.isSystemIcon(iconReference)) {
            val resId = IconMapper.getResIdFromName(iconReference)
            imageView.setImageResource(resId)
            return
        }

        // 2. 自定义图标：解析 ID
        val iconId = IconMapper.getCustomIconId(iconReference) ?: run {
            imageView.setImageResource(placeholderResId)
            return
        }

        // 3. 检查缓存
        val cacheKey = "icon_$iconId"
        val cachedBitmap = bitmapCache.get(cacheKey)
        if (cachedBitmap != null) {
            imageView.setImageBitmap(cachedBitmap)
            return
        }

        // 4. 异步加载
        scope.launch {
            val response = RetrofitClient.apiService.getIconDetail(iconId)
            if (response.isSuccessful) {
                val iconContent = response.body()?.data?.icon_content
                if (!iconContent.isNullOrEmpty()) {
                    val bitmap = withContext(Dispatchers.IO) {
                        IconMapper.loadBitmapFromBase64(iconContent)
                    }
                    bitmap?.let {
                        bitmapCache.put(cacheKey, it)
                        withContext(Dispatchers.Main) {
                            imageView.setImageBitmap(it)
                        }
                    }
                }
            }
        }
    }
}
```

**缓存策略流程图：**

```
请求加载图标
    ↓
是否系统图标?
    ├── 是 → 直接设置资源 ID，完成
    └── 否 → 解析自定义图标 ID
              ↓
         检查缓存?
              ├── 命中 → 直接使用缓存，完成
              └── 未命中 → 显示占位图
                           ↓
                      异步请求服务器
                           ↓
                      解码 Base64
                           ↓
                      存入缓存
                           ↓
                      切换主线程更新 UI
```

## 五、设计考量

### 5.1 为什么使用 Base64 存储图标？

| 方案 | 优点 | 缺点 |
|------|------|------|
| Base64 存数据库 | 部署简单，无需文件服务器 | 存储占用较大，传输较慢 |
| 图片文件服务器 | 节省数据库空间，CDN 加速 | 需要额外配置文件服务 |
| 本地资源文件 | 加载最快 | 无法自定义，需要发版更新 |

**项目选择 Base64 的原因：**
- 个人项目，规模较小
- 部署简单，无需额外服务
- 图标尺寸一般较小（经过压缩）

### 5.2 缓存策略设计

**双层缓存：**
1. **IconMapper.bitmapCache**：缓存 Base64 → Bitmap 的解码结果
2. **IconLoader.bitmapCache**：缓存网络加载的图标 Bitmap

**缓存容量：** 5MB
- 足够存储数十个图标
- 避免内存溢出
- LRU 策略自动淘汰不常用图标

### 5.3 图标引用格式设计

**格式：** `{type}:{identifier}`

| 类型 | 示例 | 解析结果 |
|------|------|----------|
| 系统图标 | `"book"` | type=system, identifier=book |
| 自定义图标 | `"custom:123"` | type=custom, identifier=123 |

**为什么不用统一的格式？**
- 系统图标是历史数据，已存在
- 保持向后兼容
- 自定义图标使用 `custom:` 前缀便于识别

### 5.4 颜色设计

**预设颜色的好处：**
- UI 风格统一
- 用户无需担心配色问题
- 避免出现刺眼的颜色
- 减少用户决策成本

## 六、注意事项

### 6.1 图片压缩

前端在上传前需要压缩图片：
- 限制最大尺寸（如 200x200）
- 压缩质量（如 80%）
- 转换为 PNG/JPEG 格式

### 6.2 缓存清理

```kotlin
// 在内存紧张时清理
IconLoader.clearCache()
```

### 6.3 删除分类的影响

删除分类时，已关联的任务不会被删除：
- 任务的 `category_id` 变为 `NULL`
- 任务变为"未分类"状态
- 在分类列表中显示在最后
