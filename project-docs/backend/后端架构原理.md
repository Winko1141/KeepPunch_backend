# 后端架构原理

## 一、架构概述

### 1.1 设计理念

后端采用经典的 **分层架构** 设计，结合 Koa 框架的中间件机制，实现高内聚、低耦合的代码结构。核心设计原则：

- **轻量级**：使用 Koa 而非 Express，减少框架开销
- **模块化**：按业务领域划分路由模块
- **Promise 优先**：全程使用 async/await 异步编程
- **连接池**：使用数据库连接池提高性能

### 1.2 分层结构

```

│                    Client (Android)                      │
```

## 二、核心组件解析

### 2.1 应用入口 (app.js)

#### 中间件配置原理

```javascript
// CORS 中间件：处理跨域请求
app.use(cors());

// JSON 中间件：统一 JSON 响应格式
app.use(json());

// BodyParser：解析请求体，支持 10MB 的大数据
app.use(bodyParser({
    jsonLimit: '10mb',
    formLimit: '10mb',
    textLimit: '10mb'
}));
```

**设计考量：**
- **CORS**：为 Android 客户端提供跨域支持
- **BodyParser 大小限制**：支持图片上传（自定义图标功能）
- **中间件顺序**：按洋葱模型执行，请求自上而下，响应自下而上

#### 路由挂载机制

```javascript
// 主路由（测试接口）
app.use(router.routes());
app.use(router.allowedMethods());

// 业务路由
app.use(taskRouter.routes());
app.use(taskRouter.allowedMethods());
// ... 其他路由
```

**allowedMethods() 的作用：**
- 自动处理 OPTIONS 请求（预检请求）
- 对不支持的 HTTP 方法返回 405 Method Not Allowed
- 对不允许的方法返回 501 Not Implemented

### 2.2 数据库层 (db/index.js)

#### 连接池原理

```javascript
const pool = mysql.createPool({
  host: 'localhost',
  user: 'root',
  password: 'root',
  database: 'keeppunch',
  waitForConnections: true,      // 等待可用连接
  connectionLimit: 10,           // 最大连接数
  queueLimit: 0                  // 无限队列
});
```

**连接池优势：**
1. **资源复用**：避免频繁创建销毁连接的开销
2. **并发控制**：限制最大连接数，防止数据库过载
3. **连接管理**：自动回收空闲连接

#### Promise API 优势

使用 `mysql2/promise` 而非回调风格的 `mysql`：

```javascript
// Promise 风格（推荐）
const [rows] = await pool.query(sql, params);

// vs 回调风格
pool.query(sql, params, (err, rows) => { ... });
```

**好处：**
- 支持 async/await，代码更清晰
- 更好的错误处理（try/catch）
- 符合现代 JavaScript 习惯

### 2.3 路由模块设计

#### 前缀路由

每个路由模块使用统一前缀：

```javascript
const router = new Router({ prefix: '/task' });
// 最终路径: /task/create, /task/list, etc.
```

**优点：**
- URL 结构清晰
- 模块边界明确
- 便于权限控制（未来扩展）

#### 统一响应格式

所有接口遵循一致的响应结构：

```javascript
{
    code: 200,        // 业务状态码
    msg: 'success',   // 消息
    data: {}          // 数据
}
```

**业务状态码约定：**
| code | 含义 |
|------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 403 | 权限不足/禁止操作 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

## 三、核心业务原理

### 3.1 任务管理原理

#### 重复类型设计

系统支持三种任务重复类型：

| repeat_type | 含义 | week_rule | target_count |
|------------|------|-----------|--------------|
| 1 | 按周重复 | 指定星期几（1-7） | 忽略 |
| 2 | 周目标 | 忽略 | 每周需要完成次数 |
| 3 | 月目标 | 忽略 | 每月需要完成次数 |

#### 任务列表筛选逻辑

当查询指定日期的任务列表时，需要进行多层过滤：

```
1. 基础过滤：status = 1（活跃状态）
        ↓
2. 分类过滤：category_id 匹配
        ↓
3. 周规则过滤：repeat_type=1 时检查 week_rule
        ↓
4. 目标计数过滤：
   - repeat_type=2：检查本周完成次数 < target_count
   - repeat_type=3：检查本月完成次数 < target_count
```

#### 软删除机制

任务使用 `status` 字段实现软删除：
- `status = 1`：活跃
- `status = 2`：终止（软删除）

**优点：**
- 保留历史数据（打卡记录仍然有效）
- 支持恢复功能
- 统计数据完整

### 3.2 打卡事务原理

#### 事务边界

打卡操作需要同时更新两个表，必须使用事务保证原子性：

```javascript
const connection = await pool.getConnection();
try {
    await connection.beginTransaction();
    
    // 1. 检查是否已打卡
    const [existing] = await connection.query(checkSql, [...]);
    if (existing.length > 0) {
        await connection.rollback();
        return;  // 已打卡，回滚
    }
    
    // 2. 插入打卡记录
    await connection.query(insertSql, [...]);
    
    // 3. 更新计数表
    await connection.query(upsertWeekCountSql, [...]);
    
    await connection.commit();  // 提交事务
    
} catch (err) {
    await connection.rollback();  // 回滚
} finally {
    connection.release();  // 归还连接
}
```

#### 为什么需要事务？

如果没有事务保护，可能出现：
- 打卡记录插入成功，但计数表没更新 → 统计错误
- 计数表更新了，但打卡记录没插入 → 数据不一致

#### 日期验证

```javascript
const targetDate = new Date(record_date);
const today = new Date();
today.setHours(0, 0, 0, 0);
targetDate.setHours(0, 0, 0, 0);

if (targetDate > today) {
    // 不能打卡未来日期
}
```

**设计原因：**
- 防止用户"提前"完成未来的任务
- 保证数据真实性

### 3.3 统计计算原理

#### 周统计

```
算法步骤：
1. 根据传入日期，计算该周的起止日期（周一到周日）
2. 查询所有活跃任务
3. 查询该周内所有打卡记录
4. 按 task_id 分组，构建 Map<taskId, Set<date>>
5. 对每个任务，检查一周七天的打卡状态
```

#### 连续天数计算

```javascript
// 遍历排序后的打卡日期，计算最长连续
for (let i = 1; i < sortedDates.length; i++) {
    const diff = Math.floor((sortedDates[i] - sortedDates[i-1]) / (1000 * 60 * 60 * 24));
    if (diff === 1) {
        tempStreak++;  // 连续，计数+1
    } else {
        maxStreak = Math.max(maxStreak, tempStreak);
        tempStreak = 1;  // 中断，重置
    }
}
```

**当前连续天数判定：**
- 最后打卡日期 = 昨天 或 今天
- 且从最后日期往前检查连续性

### 3.4 计数表优化原理

#### 预聚合设计

使用 `task_count` 表预聚合每周/每月的完成次数：

```sql
-- 插入或更新计数
INSERT INTO task_count (task_id, year, month, week, total_count)
VALUES (?, ?, ?, ?, 1)
ON DUPLICATE KEY UPDATE total_count = total_count + 1
```

**为什么不直接 count()？**
- 全表 COUNT 在数据量大时性能差
- 预聚合后查询 O(1)
- 写入时一次计算，多次读取受益

#### 适用场景

适用于：
- 读多写少的场景
- 统计报表频繁查询
- 历史数据不再变化

## 四、安全与性能考量

### 4.1 当前安全状况

| 方面 | 状态 | 说明 |
|------|------|------|
| SQL 注入 | ✅ 安全 | 使用参数化查询 |
| XSS | ⚠️ 基础 | 响应数据未做转义 |
| CSRF | ❌ 未保护 | 无 token 机制 |
| 密码存储 | ❌ 硬编码 | 数据库密码在代码中 |

### 4.2 参数化查询

项目正确使用了参数化查询防止 SQL 注入：

```javascript
// 安全写法（参数化）
const sql = `SELECT * FROM task WHERE id = ?`;
const [tasks] = await pool.query(sql, [taskId]);

// 危险写法（字符串拼接）
const sql = `SELECT * FROM task WHERE id = ${taskId}`;  // 勿用！
```

### 4.3 性能优化点

**已实现的优化：**
- ✅ 数据库连接池
- ✅ 预聚合计数表
- ✅ 按需查询字段
- ✅ 事务管理

**可改进的方向：**
- 添加 Redis 缓存热点数据
- 分页查询（当前全部返回）
- 数据库索引优化
- 查询日志监控
