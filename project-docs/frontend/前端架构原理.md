# 前端架构原理

## 一、架构概述

### 1.1 设计理念

前端采用 **单 Activity + 多 Fragment** 架构，结合 Android Jetpack 组件，实现模块化、可维护的代码结构。核心设计原则：

- **单 Activity**：使用 Navigation 组件管理 Fragment 切换
- **MVVM 模式**：数据与视图分离
- **异步优先**：使用 Kotlin Coroutines 处理异步操作
- **模块化**：按功能划分包结构

### 1.2 分层结构

```

│                        UI Layer                               │
│  ┌──────────────────────────────────────────────────────┐    │
```

## 二、核心组件解析

### 2.1 主活动 (MainActivity)

文件路径：[MainActivity.kt](file:///d:/xm/KeepPunch_frontend/app/src/main/java/com/example/keeppunch/MainActivity.kt)

#### 职责

1. **容器角色**：承载所有 Fragment
2. **导航管理**：设置底部导航和 ActionBar
3. **连接测试**：启动时测试后端连接

#### 关键代码

```kotlin
class MainActivity : AppCompatActivity() {
    
    override fun onCreate(savedInstanceState: Bundle?) {
        // 1. 测试后端连接
        testConnect()
        
        // 2. 视图绑定
        binding = ActivityMainBinding.inflate(layoutInflater)
        setContentView(binding.root)
        
        // 3. 获取 NavController
        val navHostFragment = supportFragmentManager
            .findFragmentById(R.id.nav_host_fragment_activity_main) as NavHostFragment
        val navController = navHostFragment.navController
        
        // 4. 配置 ActionBar
        val appBarConfiguration = AppBarConfiguration(
            setOf(R.id.navigation_home, R.id.statsContainerFragment)
        )
        setupActionBarWithNavController(navController, appBarConfiguration)
        
        // 5. 绑定底部导航
        binding.navView.setupWithNavController(navController)
    }
}
```

#### NavHostFragment 原理

使用 NavHostFragment 而非直接 findNavController 的原因：
- 确保在 Activity onCreate 中能正确获取 NavController
- 避免 Fragment 生命周期问题

### 2.2 网络层设计

#### 2.2.1 RetrofitClient

文件路径：[RetrofitClient.kt](file:///d:/xm/KeepPunch_frontend/app/src/main/java/com/example/keeppunch/network/RetrofitClient.kt)

```kotlin
object RetrofitClient {
    private const val BASE_URL = "http://10.78.237.209:3000/"
    
    val apiService: ApiService by lazy {
        Retrofit.Builder()
            .baseUrl(BASE_URL)
            .client(HttpClient.client)
            .addConverterFactory(GsonConverterFactory.create())
            .build()
            .create(ApiService::class.java)
    }
}
```

**设计要点：**
- **单例模式**：使用 object 关键字确保唯一实例
- **懒加载**：by lazy 延迟初始化
- **配置分离**：HttpClient 单独管理 OkHttp 配置

#### 2.2.2 ApiService

文件路径：[ApiService.kt](file:///d:/xm/KeepPunch_frontend/app/src/main/java/com/example/keeppunch/network/ApiService.kt)

```kotlin
interface ApiService {
    @GET("test")
    suspend fun test(): Response<BaseResponse>
    
    @POST("task/create")
    suspend fun createTask(@Body request: CreateTaskRequest): Response<ApiResponse<Any>>
    
    @GET("task/list")
    suspend fun getTaskList(
        @Query("date") date: String,
        @Query("category_id") categoryId: String? = null
    ): Response<ApiResponse<List<TaskItem>>>
}
```

**设计要点：**
- **suspend 函数**：支持协程异步调用
- **Response 包装**：获取 HTTP 状态码
- **泛型响应**：ApiResponse<T> 统一响应结构

#### 2.2.3 网络请求模式

```kotlin
lifecycleScope.launch {
    try {
        val response = apiService.getTaskList(dateStr, currentCategoryFilter)
        if (response.isSuccessful) {
            val body = response.body()
            if (body?.code == 200) {
                // 业务成功
            } else {
                // 业务失败
            }
        } else {
            // HTTP 失败
        }
    } catch (e: Exception) {
        // 网络异常
    }
}
```

**三层错误处理：**
1. 网络异常（Exception）
2. HTTP 错误（!response.isSuccessful）
3. 业务错误（body.code != 200）

### 2.3 Fragment 生命周期管理

#### ViewBinding 模式

```kotlin
class HomeFragment : Fragment() {
    private var _binding: FragmentHomeBinding? = null
    private val binding get() = _binding!!
    
    override fun onCreateView(...): View {
        _binding = FragmentHomeBinding.inflate(inflater, container, false)
        return binding.root
    }
    
    override fun onDestroyView() {
        super.onDestroyView()
        _binding = null  // 避免内存泄漏
    }
}
```

**为什么需要 null 检查？**
- Fragment 的生命周期长于其视图
- 配置变更时 Fragment 保留，但视图销毁重建
- 不置空会导致旧视图引用泄漏

#### LifecycleScope

```kotlin
lifecycleScope.launch {
    // 协程会在 Fragment 销毁时自动取消
}
```

**优势：**
- 自动与生命周期绑定
- 避免在销毁状态下更新 UI
- 无需手动管理取消

### 2.4 加载状态管理

#### LoadingStateHelper 模式

文件路径：[LoadingStateHelper.kt](file:///d:/xm/KeepPunch_frontend/app/src/main/java/com/example/keeppunch/util/LoadingStateHelper.kt)

**统一管理四种状态：**

| 状态 | 显示视图 | 触发场景 |
|------|----------|----------|
| Loading | ProgressBar | 请求发起时 |
| Content | RecyclerView | 数据加载成功 |
| Empty | 空状态布局 | 数据为空 |
| Error | 错误状态布局 | 请求失败 |

**使用方式：**
```kotlin
loadingStateHelper = LoadingStateHelper(
    contentView = binding.scrollContent,
    loadingView = binding.layoutLoading.root,
    emptyView = binding.layoutEmpty.root,
    errorView = binding.layoutError.root
)

// 切换状态
loadingStateHelper.showLoading()
loadingStateHelper.showContent()
loadingStateHelper.showEmpty(message = "暂无数据")
loadingStateHelper.showError(message = "加载失败", onRetry = { loadData() })
```

### 3.1 日历组件原理

#### Kizitonwose Calendar

项目使用 `com.kizitonwose.calendar:view:2.5.0` 库实现日历功能。

#### 周视图配置

```kotlin
calendarView.setup(
    startDate,      // 最早显示日期
    endDate,        // 最晚显示日期
    DayOfWeek.MONDAY // 一周起始日
)
calendarView.scrollToDate(today)  // 滚动到今天
```

#### DayBinder 模式

```kotlin
calendarView.dayBinder = object : WeekDayBinder<DayViewContainer> {
    override fun create(view: View): DayViewContainer {
        return DayViewContainer(view)
    }
    
    override fun bind(container: DayViewContainer, data: WeekDay) {
        // 绑定数据到视图
        val date = data.date
        container.tvDate.text = "$month.$day"
        
        // 样式切换
        if (date == selectedDate) {
            container.root.setBackgroundResource(R.drawable.bg_calendar_selected)
        } else if (date == today) {
            container.root.setBackgroundResource(R.drawable.bg_calendar_today)
        }
    }
}
```

**局部刷新优化：**
```kotlin
calendarView.notifyDateChanged(oldDate)
calendarView.notifyDateChanged(newDate)
```

只刷新变化的日期单元格，而非整个日历。

### 3.3 提醒通知原理

#### AlarmManager 机制

文件路径：[NotificationHelper.kt](file:///d:/xm/KeepPunch_frontend/app/src/main/java/com/example/keeppunch/util/NotificationHelper.kt)

**核心流程：**

```
任务创建/更新
    ↓
NotificationHelper.scheduleTaskReminder()
    ↓
根据 reminder_type 选择调度策略
    ├── 单次提醒：指定日期时间
    ├── 每周提醒：按周规则重复
    └── 每日提醒：每天固定时间
    ↓
AlarmManager.setExactAndAllowWhileIdle()
    ↓
到达时间 → AlarmReceiver.onReceive()
    ↓
构建 Notification 并显示
```

#### 提醒类型映射

| reminder_type | 调度方式 | 使用字段 |
|--------------|----------|----------|
| 1 | 单次 | reminder_date |
| 2 | 每周 | reminder_week_rule |
| 3 | 每日 | 无 |

#### 向后兼容逻辑

```kotlin
when (reminderType) {
    1 -> scheduleOnceReminder(...)
    2 -> scheduleReminderWeekly(...)
    3 -> scheduleDailyReminder(...)
    else -> scheduleBackwardCompatibleReminder(...)
}

// 兼容模式：根据任务的 repeat_type 判断
private fun scheduleBackwardCompatibleReminder(...) {
    when (task.repeat_type) {
        1 -> scheduleWeeklyReminders(...)  // 周重复任务
        else -> scheduleDailyReminder(...)   // 其他默认每日
    }
}
```

#### RequestCode 生成

```kotlin
private fun getRequestCode(taskId: Int, dayOfWeek: Int): Int {
    return taskId * 10 + dayOfWeek
}
```

**设计原理：**
- 每个任务最多 8 个提醒（每天 + 7 个周几）
- taskId * 10 确保不同任务不冲突
- 日提醒用 dayOfWeek=0

#### 精确唤醒

```kotlin
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.M) {
    alarmManager.setExactAndAllowWhileIdle(
        AlarmManager.RTC_WAKEUP,
        calendar.timeInMillis,
        pendingIntent
    )
} else {
    alarmManager.setExact(...)
}
```

**为什么需要 setExactAndAllowWhileIdle？**
- Android 6.0+ 引入 Doze 模式
- 标准 setExact 在 Doze 模式下可能延迟
- setExactAndAllowWhileIdle 即使在省电模式也能精确触发

## 四、安全与性能考量

### 4.1 当前安全状况

| 方面 | 状态 | 说明 |
|------|------|------|
| 后端地址 | ⚠️ 硬编码 | BASE_URL 在代码中 |
| 数据传输 | ❌ HTTP | 未使用 HTTPS |
| 本地存储 | ⚠️ 基础 | 未加密敏感数据 |

### 4.2 性能优化点

**已实现的优化：**
- ✅ ViewBinding 替代 findViewById
- ✅ Coroutines 异步网络请求
- ✅ 局部刷新（notifyDateChanged）
- ✅ 延迟初始化（lazy）

**可改进的方向：**
- 添加 RecyclerView DiffUtil
- 图片缓存（自定义图标）
- 分页加载（大量任务时）
- 数据本地缓存（Room）
